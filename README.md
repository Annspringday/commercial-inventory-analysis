# Кейс: Оптимизация складских запасов и выявление неликвидных активов

**Стек технологий:** SQL (MySQL), Power BI, DAX, ETL / Моделирование данных.  
**Бизнес-домен:** Логистика, Управление запасами (Inventory Management), Коммерческая аналитика.

---

## 1. Описание бизнес-задачи

**Заказчик:** Коммерческий директор торговой компании.  
**Проблема:** На складе компании наблюдается дефицит свободных площадей, а оборачиваемость по ряду категорий падает. Существует подозрение, что значительная часть оборотных средств заморожена в неликвидных товарах («висяках»), которые занимают место, но не генерируют выручку.

**Цель проекта:** * Выявить позиции, закупки и продажи по которым прекратились или осуществлялись очень давно.
* Рассчитать точный объем «замороженных» финансовых средств в ценах себестоимости.
* Определить физическую долю залежалого товара от общего объема склада.
* Построить интерактивный дашборд в Power BI для топ-менеджмента, чтобы принять операционные решения (запуск акций, распродаж, списание).

---

## 2. Структура и моделирование данных

Анализ базируется на трех исходных таблицах, выгруженных из ERP-системы компании:
1. `products` (Справочник товаров): `product_id`, `product_name`, `category`, `price`.
2. `stock` (Текущие остатки): `product_id`, `quantity_in_stock`.
3. `sales` (Журнал транзакций продаж): `sale_id`, `product_id`, `quantity_sold`, `sale_date`.

В Power BI была реализована классическая схема моделирования **«Звезда» (Star Schema)**. Таблица `products` выступает в роли главного справочника (сторона `1`), связываясь с таблицей фактов `sales` (сторона «многие», связь `*:1`) и таблицей остатков `stock` (связь `1:1`) по ключевому полю `product_id`.

## 3. Этап SQL-анализа (Выгрузка и расчеты)

Для первичного выявления залежалого товара был написан комплексный SQL-запрос. Чтобы избежать критической ловушки дублирования строк при объединении таблиц (из-за связи «один ко многим» с таблицей продаж), агрегация сумм и расчет замороженных средств реализованы с использованием функции `MAX()` и группировки.

Также в запрос был внедрен независимый подзапрос для расчета доли штук конкретного товара от суммарного количества всех позиций на складе:

```sql
SELECT 
    p.product_id,
    p.product_name,
    p.category,
    MAX(st.quantity_in_stock) AS stock_qty,
    MAX(s.sale_date) AS last_sale_date,
    SUM(st.quantity_in_stock * p.price) AS frozen_money,
    -- Расчет процента от общего объема склада через подзапрос
    ROUND(MAX(st.quantity_in_stock) / (SELECT SUM(quantity_in_stock) FROM stock) * 100, 4) AS pct_of_total_stock
FROM products p
LEFT JOIN stock st ON p.product_id = st.product_id
LEFT JOIN sales s ON p.product_id = s.product_id
GROUP BY p.product_id, p.product_name, p.category
HAVING last_sale_date < '2026-03-01' OR last_sale_date IS NULL
ORDER BY frozen_money DESC;
```

### Результаты выполнения SQL-запроса:
| ID | Товар | Категория | Остаток | Последняя продажа | Доля склада | Заморожено |
|---|---|---|---|---|---|---|
| 103 | Ботинки зимние | Обувь | 80 шт | 2025-11-20 | 16.74% | 719 920 ₽ |
| 108 | Джинсы зауженные | Одежда | 65 шт | 2026-02-14 | 13.60% | 259 935 ₽ |
| 110 | Кепка спортивная | Аксессуары | 95 шт | NULL (нет) | 19.87% | 94 905 ₽ |

**Ключевой инсайт SQL-этапа:** Суммарно три эти позиции составляют **50.21%** (более половины!) от всего физического объема склада. При этом продажи по ним не было уже несколько месяцев (или не было вообще, как по кепкам).

## 4. Этап визуализации и DAX-моделирования (Power BI)

Чтобы динамически управлять расчетами без "жесткого" кодирования в таблицах, в Power BI были разработаны следующие меры на языке DAX:

### 1. Определение даты последней продажи по каждому товару:
```dax
Дата последней продажи = MAX(sales[sale_date])

### 2. Базовая мера стоимости текущего остатка (динамический расчет стоимости):
```dax
Замороженные деньги = SUMX(stock, stock[quantity_in_stock] * RELATED(products[price]))

### 3. Изолированная мера для KPI-карточки (фиксация неликвидных активов):
Поскольку одиночные KPI-карточки работают в глобальном контексте фильтрации, стандартные визуальные фильтры могут некорректно обрабатывать пустые значения (`BLANK`). Для точного отображения агрегированного итога по "висякам" была написана продвинутая мера с функцией `CALCULATE` и внутренним табличным фильтром:

```dax
Замороженные деньги ВИСЯКИ = 
CALCULATE(
    [Замороженные деньги],
    FILTER(
        products,
        [Дата последней продажи] <= DATE(2026, 3, 1) || ISBLANK([Дата последней продажи])
    )
)
> **Итоговый результат в KPI-карточке:** `1.07 Млн ₽`[cite: 1, 2]

Разработанный интерактивный отчет включает в себя три ключевых визуальных компонента:[cite: 1, 2]
* **KPI-Карточка:** Выводит главную верхнеуровневую метрику для руководства — общую сумму потерь (**1.07 млн руб.**)[cite: 1, 2].
* **Линейчатая гистограмма:** Показывает распределение остатков неликвида в штуках (лидируют спортивные кепки — **95 шт.**, и зимние ботинки — **80 шт.**)[cite: 1, 2].
* **Круговая диаграмма (Donut Chart):** Отражает долю структуры замороженных денег в денежном эквиваленте. Из неё отчетливо видно, что основная масса заблокированного капитала (**66.98%**) приходится на категорию *«Обувь»* (Ботинки зимние кожаные — **719,92 тыс. руб.**)[cite: 1, 2].

### Рекомендации для коммерческого департамента:

1. **Категория «Обувь» (Ботинки зимние):** Заморожено **719.9 тыс. руб.** Требуется объявить сезонную тотальную распродажу со скидкой до *30-40%*. Деньги необходимо вытащить из оборотного тупика до окончания текущего квартала.
   
2. **Категория «Аксессуары» (Кепки спортивные):** Занимают почти **20%** физического пространства склада, при этом имеют нулевые продажи. Рекомендуется использовать их в качестве подарка (акция *«cross-sell»*) при покупке высокомаржинальных товаров (например, беговых кроссовок) для стимуляции основных продаж и оперативного освобождения складских площадей.
   
3. **Категория «Одежда» (Джинсы зауженные):** Заморожено **259.9 тыс. руб.** Рекомендуется запустить стимулирующую акцию формата *«1+1=3»* или *«вторая пара джинсов со скидкой 50%»*.
